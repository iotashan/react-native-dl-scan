# iOS build findings — fast-tflite native-orchestration integration

Empirical results from actually building the example on this machine (Xcode 26.2,
iOS Sim, RN 0.81 / Expo SDK 54). This is what a real compile surfaced that could
not be known offline.

## What the build VALIDATED (works)
- `example` install + `expo prebuild -p ios` + `pod install`: **91 pods**, incl.
  `react-native-fast-tflite (3.0.1)` + `TensorFlowLiteC (2.17.0)` + VisionCamera.
  No monorepo-shadowing problem (the `react-native-dl-scan` → `..` node_modules
  symlink is required for autolinking; metro already documents it).
- The DlScan pod autolinks and the Nitro cross-module bridge generates.
- After adding `s.dependency "react-native-fast-tflite"` + the NitroTflite
  modulemap flag, the `<NitroTflite/HybridTfliteModelSpec.hpp>` include **does
  resolve** and Expo modules build — i.e. the model handoff compiles in principle.
- The C++ core (preprocess/decode/C-ABI/orchestrator) compiles into the pod
  (cpp/detect now in `source_files`).

## The BLOCKER (well-diagnosed; needs a decision)
nitrogen emits, in the generated `HybridDlScanSpec.hpp`:
```cpp
#include <NitroTflite/HybridTfliteModelSpec.hpp>
```
But fast-tflite vends that header at
`Pods/Headers/Public/react-native-fast-tflite/HybridTfliteModelSpec.hpp`
(**pod name**), and only symlinks its umbrella+modulemap under
`Pods/Headers/Public/NitroTflite/` — NOT the header itself. (NitroImage, by
contrast, symlinks its actual C++ headers under `Public/NitroImage/`, so literal
`<NitroImage/x.hpp>` lookup works — that's the asymmetry.)

So literal header lookup for `<NitroTflite/...>` fails. The `-fmodule-map-file`
workaround makes it resolve as a module import (the umbrella re-exports the
header) BUT passing an explicit modulemap **suppresses clang's implicit modulemap
discovery** for the other modules compiled in the same target — cascading into
`could not build module 'ExpoModulesCore'` (Swift-flag scope) and
`'NitroModulesSpec/NitroModulesSpec.h' file not found` (C/C++-flag scope). Whack-a-mole.

## Fix options (pick one in a follow-up)
1. **Switch to JS/worklet orchestration (recommended).** Don't reference
   `TfliteModel` in our Nitro spec at all → no cross-module C++ include, no
   header problem. JS loads the model (`loadTensorflowModel`) and calls
   `runSync`; it passes the input/output ArrayBuffers to our C-ABI
   (`dlscan_preprocess_field`/`dlscan_decode_field`) exposed as plain Nitro
   methods. The C++ core (preprocess/decode/C-ABI) is unchanged and already
   tested (282). This sidesteps the entire packaging issue. NOTE: this revisits
   the earlier "native orchestration" decision — the build empirically shows
   native orchestration carries a real, gnarly integration cost here.
2. **Upstream fix**: get fast-tflite to vend its C++ headers under `NitroTflite/`
   (like NitroImage) — then literal `<NitroTflite/...>` resolves with no flags.
3. **Podfile post_install header symlink**: symlink the fast-tflite C++ headers
   under `Pods/Headers/Public/NitroTflite/` after each install. Works but fragile
   (regenerated by prebuild/pod install; needs a config plugin to persist).

## Also still required to RUN (not just compile)
- The native call-site shims invoking the orchestrator (or, under option 1, the
  JS worklet calling runSync + the C-ABI Nitro methods).
- Trained NanoDet `.tflite` weights (cloud GPU) for the field detector to
  produce real detections. DocAligner ships and would run once wired.

## SECOND blocker (after the header symlink fixed the first) — CONCLUSIVE
With the C++ header resolved (symlink fast-tflite's headers under
`Pods/Headers/Public/NitroTflite/`, like NitroImage does — no modulemap flag, no
cascade), the build advanced all the way to compiling our own Swift and hit:
```
ios/HybridDlScanIOS.swift:406: error: cannot find type 'HybridTfliteModelSpec' in scope
```
Only the HAND-WRITTEN setter errors — nitrogen's *generated* `HybridDlScanSpec.swift`
uses the identical `(any HybridTfliteModelSpec)` and compiles. Root cause: fast-tflite
is `HybridObject<{ios:'c++'}>`, so it ships **no Swift protocol** for `TfliteModel`
(verified: no `protocol/typealias/class HybridTfliteModelSpec` anywhere in the
package or build). The type is only usable from C++/generated-bridge contexts, not as
a Swift-typed stored member in hand-written code.

## VERDICT (build-validated, not hypothesized)
Native orchestration fights fast-tflite's C++-only packaging at BOTH the C++ header
layer AND the Swift type layer. **Switch to JS/worklet orchestration** — it removes
the `TfliteModel` reference from our Nitro spec entirely, so neither blocker exists:
- Our Nitro spec exposes plain C-ABI-style methods (ArrayBuffer in/out) — no
  cross-module HybridObject type.
- JS (or a VC frame-processor worklet) loads the model via `loadTensorflowModel`,
  calls `runSync`, and threads the tensors through our methods to the C-ABI
  (`dlscan_preprocess_field`/`dlscan_decode_field` + docaligner). The tested C++ core
  (282) is unchanged.
This reverses the earlier "native orchestration" pick — but that pick was made before
the build revealed its real, multi-layer cost. The header-symlink workaround does let
native orchestration limp forward on the C++ side, but the Swift-type wall makes it a
dead end without also hand-writing/patching a Swift shim for a c++-only object.

## PIVOT EXECUTED + build 6 result
The JS-orchestration pivot is implemented and committed:
- Nitro spec: `TfliteModel` setters → ArrayBuffer C-ABI methods
  (preprocessFieldInput/decodeFieldOutput/preprocessDocAlignerInput/decodeCorners
  + FieldDetectionSpec). nitrogen regenerated with **ZERO** NitroTflite refs; tsc+eslint clean.
- iOS Swift: real best-effort impls via Cxx interop to detect_c (marked UNVALIDATED).
- Android Kotlin: dead setters removed; the 4 methods are documented stubs (JNI to
  detect_c is the remaining Android task; legacy path still runs).
- Podspec: fast-tflite dependency + header path removed (no native NitroTflite coupling).

**Build 6 (after the pivot):** `grep NitroTflite` → 0 hits — the original blocker is
GONE. The build now fails earlier on an UNRELATED issue: `react-native-svg`'s ObjC
headers can't find `React/UIView+React.h` / `React/RCTConvert.h` (RN 0.81 prebuilt-React
header layout). svg compiled fine in builds 1–5; the post-pivot Pods regen shuffled
build ordering and exposed it. This is environmental (Expo SDK 54 / RN-svg / prebuilt
React), NOT the DlScan refactor — DlScan wasn't even reached. Likely fixes: a clean
`pod install`/DerivedData wipe, or pinning react-native-svg's React header search path.
Once cleared, the build reaches DlScan and validates the Swift C-ABI impls (then the
Kotlin JNI impls + a trained NanoDet `.tflite` remain to RUN the field detector).
